# 神苑スタッフ 参加申込・事前決済・受付管理システム 要件定義書（v4 全確定版）

作成日：2026-06-07
ステータス：要件確定（残りは詳細仕様・ベンダー選定のみ）
関連資料：画面設計書 v1

---

## 1. システムの目的・概要

神苑が運営する **複数のイベントを継続的に** 扱い、全国の拠点に所属する神苑スタッフが、参加を **スマホから申込・事前決済** し、運営側が **申込内容・当日受付を効率的かつ漏れなく管理** するためのシステム。

申込は「個人単位」で行い、拠点の **代表者がまとめて確定**。確定後に各個人が事前決済する（確定後決済型）。申込締切は **毎月25日**。想定規模は **1イベントあたり数百人**。参加者・代表者・管理者はいずれも **ログイン制**。

---

## 2. 登場人物（アクター）

| アクター | 役割 |
| --- | --- |
| 参加者（個人） | 神苑スタッフ。**自己登録**、または代行入力時に発行されたアカウントでログインし、自分の情報を入力。確定後に各自で事前決済する |
| 代表者 | 拠点ごとに設定。拠点の名簿を確認・確定、必要に応じて全員分を代行入力。申込内容の修正も可 |
| 管理者 | 運営側。イベント・拠点・代表者・フォーム項目の設定、申込全体の閲覧・修正・削除・返金、CSV出力 |
| 受付担当 | 当日、来場者の出欠をチェック（管理者・代表者が兼任の場合あり） |

※ 申込内容の修正・削除ができるのは **代表者と管理者のみ**（参加者本人は不可）。
※ 代表者は **拠点ごとに設定** できる。

---

## 3. データ構造

複数イベントを継続運用するため、最上位に「イベント」を置く。**金額・決済・キャンセルはすべて参加者（個人）単位**で持つ。

```
イベント（複数を継続運用／管理者が作成・複製可／締切は毎月25日）
 └─ 申込（拠点単位・代表者が確定）  ←── 拠点マスタ（全国の拠点／イベント横断で共有・拠点ごとに代表者を設定）
     └─ 参加者（個人）× 複数名
         ├─ ステータス：申込中 / 確定 / 支払済 / キャンセル
         ├─ カスタム入力項目（個人ごと）
         └─ 金額（個人ごと）
```

- 決済手段がカード・PayPayのみ（即時決済）のため「入金待ち」状態は不要。
- 拠点マスタはイベントをまたいで共有（毎回作り直さない）。
- 合計金額は「支払済」の参加者を **拠点ごと・イベントごと・全体で自動集計**。
- キャンセルされた人は集計から自動的に除外。

## 4. 申込〜決済フロー（確定後決済型）

```
[0] アカウント作成（自己登録、または代表者の代行入力時に発行）
        ↓
[1] 申込登録（仮登録・未決済）
     ・自己申込：個人がログインし、拠点を選択して自分で入力
     ・代行入力：代表者が拠点の全員分を入力
     ※この段階のキャンセルは「名簿から外すだけ」で完結
        ↓
[2] 代表者が拠点名簿を確認・確定（毎月25日の締切までに／キャンセル者を除外）
        ↓
[3] 確定者へ決済依頼（個人ごとに決済リンクを「メールで」送付）
        ↓
[4] 各個人がログインして事前決済（カード／PayPay、いずれも即「支払済」）
     ・未決済者は代表者／管理者がリマインド
        ↓
[5] 当日受付（人単位の出欠＋拠点ごとの集計・未受付の可視化）
```

## 5. 機能要件

### 5-1. 参加者（ユーザー）側 ― スマホファースト

- **自己登録**でアカウント作成（氏名・メール・パスワード・所属拠点）。代行入力で発行された場合は初回ログイン時にパスワード設定
- ログインしてイベント・拠点を選択し参加申込（拠点はマスタから選択）
- カスタム入力項目に沿った情報入力、**合計金額の自動計算・表示**
- 確定後、メールで届く決済リンクからログインして事前決済（カード／PayPay）
- 申込・決済完了時の通知

### 5-2. 代表者の機能

- 拠点メンバーの **代行入力**（全員分まとめて入力／入力時にアカウント発行）
- 拠点名簿の確認、キャンセル者の除外、**名簿の確定**（毎月25日締切）
- 申込内容の修正、決済状況の確認・未決済者へのリマインド

### 5-3. 管理者の機能

- **イベント管理**：作成・編集・**複製**、締切（**デフォルト毎月25日**）・対象拠点・利用フォーム・定員の設定
- **拠点マスタ管理**：拠点の登録・編集・削除、**拠点ごとの代表者設定**
- **フォーム項目のカスタマイズ**：追加／編集／削除／並び替え、項目タイプ（テキスト・選択・数値・日付）、必須/任意、**金額への連動**（選択肢・数量に単価を紐づけ合計へ反映）
- **申込一覧表示**：検索・絞り込み（イベント・拠点・ステータス）・並び替え、拠点単位＋人単位明細
- **CSV出力**（人単位の明細）
- **申込内容の修正・削除**、**キャンセル／返金**管理
- アカウント・権限管理

### 5-4. 当日受付・出欠管理

- 人単位の出欠（来場済／未来場）を記録
- **未受付者がひと目でわかる**（漏れ防止）
- 受付効率化：**QRコード読み取り と 氏名検索 の両対応**
- 拠点ごと・全体の受付状況をリアルタイム集計（来場数／総数）
- 受付画面から「当日キャンセル」をワンタップ記録（→ 集計から除外）

## 6. 決済・返金の運用方針

### 6-1. 決済手段（2種・いずれも即時決済）

| 手段 | 確定タイミング |
| --- | --- |
| クレジットカード | 即時「支払済」 |
| PayPay | 即時「支払済」 |

- 両手段を扱える国内向け決済代行を選定（返金も代行経由で完結）。

### 6-2. 再決済を複雑にしない方針

**お金の操作は「決済」と「返金」の2つだけに統一。** 差額の追加課金は作らない。

- 決済前の修正 → 金額が変わっても自由（確定時に最終額で決済依頼するだけ）
- 決済後・金額が変わらない修正 → そのまま修正、決済操作なし
- 決済後・金額が変わる修正 → 「キャンセル（返金）→ 新内容で申込・再決済」に一本化

### 6-3. キャンセル・返金ポリシー

- **イベント当日のキャンセル → 返金なし**
- **前日までのキャンセル → 全額返金**
- 当日キャンセルは受付画面からワンタップ記録 → 集計から除外
- カード・PayPayとも返金は決済代行経由で処理（手作業の振込戻しは不要）

### 6-4. 通知

- 通知手段は **メールと LINE の両対応**
- **決済リンクは確実性のため必ずメールで送付**（LINE併用も可）

---

## 7. 非機能要件

- ユーザー画面はスマホファースト。管理画面はPC・タブレット併用
- 数百人規模の申込・決済集中に耐える
- 個人情報・決済情報のセキュリティ（決済情報は決済代行に預け自前保持しない）
- バックアップ、誤削除・誤操作対策
- 操作ログ（誰がいつ確定・修正・返金・受付したか）

---

## 8. 確定済みの主要決定事項（全17項目）

1. 神苑は組織であり施設。対象は神苑スタッフ。
2. 複数イベントを継続運用（イベント管理機能が必要）。
3. 想定規模は1イベントあたり数百人。
4. 決済手段はクレジットカードと PayPay の2種（即時決済）。
5. 受付はQRコード読み取りと氏名検索の両対応。
6. 申込は拠点ごと・個人単位で管理。代表者が名簿を確定（二段構え）。
7. 代表者は拠点ごとに設定できる。
8. 参加者・代表者・管理者はいずれもログイン制。
9. 参加者アカウントは自己登録または代行入力時に発行。
10. 支払者は参加者個人。決済タイミングは確定後決済型。
11. 入力は自己申込・代行入力の両対応。拠点は事前登録マスタから選択。
12. 申込締切は毎月25日（イベント作成時のデフォルト）。
13. イベントは複製作成が可能。
14. 申込内容の修正・削除は代表者と管理者のみ。
15. 再決済は「決済／返金の2操作」に統一（差額課金は作らない）。
16. 返金ポリシー：イベント当日は返金なし、前日までは全額返金。
17. 通知はメール＋LINE。決済リンクは必ずメールで送付。
