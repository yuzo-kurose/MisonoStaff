# 設計追補：複数イベント参加・1スキャン一括受付・まとめ決済

作成日：2026-06-07
前提：要件定義書 v4／データ定義書 v2.1
背景：**同日に複数イベントが開催**され、**1人が複数イベントに参加**しうる。当日受付・申込・決済の煩雑さを減らし、ユーザーフレンドリーにする。

決定事項（2026-06-07）：
- まとめ決済：**採用**（1回のStripe Checkoutで複数イベント分）
- 開催回(occasion)エンティティ：**導入しない**（`event_date`＋`venue` で束ねる）

---

## 1. 当日受付：1スキャンで複数イベントを一括受付 ★中核

### 課題
1人が同日2イベントに参加すると、従来は受付2回・QR2枚になり煩雑。

### 設計
- **QRは「人」に紐づく**：`profiles.checkin_token`（UUID）をQR化。イベント別QRにしない。各参加者は自分のマイページで1枚のQRを提示。
- **受付画面は「日付（＋会場）」スコープ**：受付担当が当日の日付（既定=今日）と、必要なら会場を選択。
- **フロー**：
  1. QRスキャン（または氏名検索）
  2. RPC `checkin_candidates(token, date, venue)` で **その人の当日参加イベント一覧**＋各受付状況を取得
  3. 一覧を表示（既定で全イベントにチェック）。受付済みはバッジ表示
  4. **「まとめて受付」1タップ** → RPC `batch_check_in(participant_ids[], 'qr')` で一括 `checked_in`
  5. 完了トースト（「3件 受付完了」等）
- **個別対応**：特定イベントだけ欠席ならチェックを外す。当日キャンセルは `mark_day_cancelled(participant_id)` を行ごとにワンタップ。
- **集計は従来どおりイベント単位**：`attendances` は participant 単位で保持するため、イベント別・拠点別の来場数／未受付の可視化は維持される。

### 関連オブジェクト
- 列：`profiles.checkin_token`、`events.venue`
- RPC：`checkin_candidates` / `batch_check_in` / `mark_day_cancelled`（`supabase/migrations/20260607000004_reception_rpc.sql`）

---

## 2. まとめ申込：複数イベントを1フローで

### 設計（DB変更なし。画面/アプリ層で実現）
- イベント選択画面で、申込可能な（公開中・締切前・自拠点が対象の）イベントを一覧表示し、**複数チェック**して一括申込。
- **共通プロフィール（氏名・所属拠点）は profiles を再利用**し再入力させない。
- 各イベントの**カスタム項目のみ**入力。同名・同種の項目は**前回/他イベントの回答を初期値**として引き継ぐ（コピー）。
- 確定（代表者）：拠点名簿を**日付単位でまとめて確認・確定**できるUIにする（同日イベントを横断表示）。
- データ上は従来どおり、選択イベントごとに `applications`(event×branch) を find-or-create し、`participants` を作成。

### ポイント
- 「開催回」テーブルを作らない代わりに、UIは `event_date`（＋`venue`）で同日イベントをグルーピング表示する。

---

## 3. まとめ決済：複数イベントを1回の支払いに

### 設計
- 確定後、その人の**未決済 participants をまとめて** 1つの `payment_groups` を作成。
- **Stripe Checkout Session を1つ生成**し、`line_items` にイベントごとの明細（イベント名 × 金額）を並べる。`payment_method_types: ['card','paypay']`。
- 各 participant に対応する `payments`（按分行）を作成（`amount` はそのイベント分）。
- 決済リンク（Checkout URL）は**1通のメール**で送付。
- **Webhook `checkout.session.completed`**：`payment_groups.status='completed'`／`paid_at`／`method` を更新し、配下 `payments` を `completed`、対象 `participants` を `paid` に。冪等性は `stripe_payment_intent_id` UNIQUE で担保。

### 返金（イベント単位の部分返金）
- 1イベントだけキャンセル → その `payments` 分を **同一 PaymentIntent への部分返金**（Stripe Refund, amount=その按分額）。
- `refunds` 記録、`payments.status='refunded'`／`refunded_amount` 更新、`participants.status='cancelled'`。
- 全イベント返金で `payment_groups.status='refunded'`。
- 返金ポリシーは要件どおり：**当日=返金なし／前日まで=全額**（`events.event_date` と当日比較）。

### まとめ決済の確定タイミング（運用上の注意）
- 複数イベントの**確定タイミングがずれる**場合がある（代表者が別々に確定）。
- 方針：**「その時点で確定済み・未決済」の participants をまとめて決済依頼**。後から別イベントが確定したら、その分で**新たな payment_group** を作る（＝人によっては決済が複数回に分かれることを許容）。
- これにより「全イベント確定まで決済できない」ロックを避ける。同日分が同時確定なら自然に1回にまとまる。

## 4. 影響まとめ

| 項目 | 変更 |
| --- | --- |
| `profiles` | `checkin_token` 追加 |
| `events` | `venue` 追加 |
| 決済 | `payment_groups` 新設／`payments` を按分行に変更／`refunds` は据え置き（payment_id 参照先の意味が「イベント按分」に） |
| RPC | `checkin_candidates` / `batch_check_in` / `mark_day_cancelled` |
| 申込・名簿確定 | DB変更なし（UI/アプリ層で複数イベント一括化） |

## 5. 残課題

- まとめ決済の按分明細（Checkout の line item 表記、領収書の体裁）
- 同日内で会場をまたぐ参加がある場合の受付スコープ運用（会場フィルタの既定値）
- 前回回答コピーの「同一項目」判定ルール（label一致 or マスタ項目ID）
